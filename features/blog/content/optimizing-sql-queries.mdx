# How I Optimized SQL Queries to Cut Dashboard Load Times by 35%

When I took over a government land records dashboard, some report queries were taking 8-12 seconds. With 10,000+ daily queries, this was unacceptable. Here's the systematic approach that brought it down to under 2 seconds.

## Step 1: Measure Before You Touch Anything

The first rule of optimization: never guess. Run `EXPLAIN ANALYZE` on every slow query.

```sql
EXPLAIN ANALYZE
SELECT l.district, l.taluk, COUNT(*) as total_transfers,
       SUM(CASE WHEN l.status = 'approved' THEN 1 ELSE 0 END) as approved
FROM land_records l
JOIN owners o ON l.owner_id = o.id
WHERE l.transfer_date BETWEEN '2024-01-01' AND '2024-12-31'
GROUP BY l.district, l.taluk
ORDER BY total_transfers DESC;
```

The output told me immediately: **full table scan on `land_records` (2.3M rows), no index on `transfer_date`**.

## Step 2: Index the Right Columns

Missing indexes were the biggest culprit.

```sql
-- Compound index for the most common filter pattern
CREATE INDEX idx_records_date_status ON land_records(transfer_date, status);

-- Index for join performance
CREATE INDEX idx_records_owner ON land_records(owner_id);
```

Query time: 8.2s → 1.9s. Just from indexes.

## Step 3: Rewrite N+1 Queries

The dashboard had a classic N+1 problem in the approval workflow endpoint:

```javascript
// BAD: N+1 — one query per record
const records = await db.query('SELECT * FROM land_records WHERE district = ?', [district])
for (const record of records) {
  record.owner = await db.query('SELECT * FROM owners WHERE id = ?', [record.owner_id])
}
```

```javascript
// GOOD: Single JOIN
const records = await db.query(`
  SELECT l.*, o.name as owner_name, o.contact
  FROM land_records l
  JOIN owners o ON l.owner_id = o.id
  WHERE l.district = ?
`, [district])
```

For a district with 500 records, this went from 501 queries to 1.

## Step 4: Avoid SELECT *

Every `SELECT *` was pulling columns the dashboard never used. Being specific matters at scale:

```sql
-- Before: pulling 23 columns including BLOBs
SELECT * FROM land_records WHERE ...

-- After: only what the dashboard needs
SELECT id, survey_number, district, taluk, status, transfer_date
FROM land_records WHERE ...
```

## Step 5: Aggregate at the Database Level

The API was fetching raw rows and aggregating in JavaScript. Always push aggregation to the DB:

```sql
-- Let MySQL do the heavy lifting
SELECT
  district,
  COUNT(*) as total,
  SUM(status = 'pending') as pending_count,
  SUM(status = 'approved') as approved_count,
  AVG(DATEDIFF(approved_date, submitted_date)) as avg_processing_days
FROM land_records
WHERE transfer_date >= DATE_SUB(NOW(), INTERVAL 30 DAY)
GROUP BY district;
```

## Results

| Metric | Before | After |
|--------|--------|-------|
| Report load time | 8.2s | 1.8s |
| Queries per page load | 47 | 8 |
| Peak DB CPU | 85% | 28% |

## Key Principles

1. **EXPLAIN ANALYZE first** — always know where time is spent
2. **Index filter + join columns** — compound indexes for common patterns
3. **Kill N+1 queries** — JOINs beat loops every time
4. **Aggregate in SQL** — databases are faster at math than your application
5. **Select only what you need** — `SELECT *` is a lazy tax

The same techniques apply across MySQL, PostgreSQL, and Oracle. Profile first, then optimize with purpose.
